#### 2.2 新增 `prompts/experience_analyzer.md`——从原材料中提取经验

这个文件用于从聊天记录、文档、邮件等原材料中自动识别和提取经验案例：

```markdown
# 经验提取分析 Prompt

## 任务

你将收到 {name} 的原材料。从中识别并提取他解决问题的完整案例。

**与 work_analyzer 的分工：**
- work_analyzer 提取"他怎么工作"——规范、流程、偏好（静态能力）
- experience_analyzer 提取"他解决过什么问题"——具体案例、判断过程、经验教训（动态经验）

**原则：一个完整的问题解决过程 = 一条经验。碎片化的技术观点归入 work_analyzer 的经验知识库。**

---

## 识别信号

以下内容模式暗示一条经验记录：

**故障/问题相关：**
- "当时出了个问题……" / "线上挂了" / "排查了一下发现……"
- "之前踩过这个坑" / "上次也是这样"
- 包含时间线的排查描述（先……然后……最后……）
- 包含监控截图、报警信息的讨论

**方案决策相关：**
- "我们对比了 A 和 B 方案" / "最后选了……因为……"
- "本来想用 X，但是考虑到……"
- 评审会上的方案讨论和最终结论

**复盘/总结相关：**
- "复盘一下" / "总结几点"
- "以后遇到这种情况……"
- "早知道应该……"

---

## 提取规则

对每个识别到的经验案例，尽量填充以下字段：

1. **事件摘要**：一句话概括（必须提取）
2. **背景**：从上下文推断（涉及的系统、业务、时间）
3. **问题描述**：具体出了什么事
4. **分析/排查过程**：他做了什么步骤（按时间顺序）
5. **方案与决策**：最终选了什么方案，为什么
6. **结果**：如果提到了结果/效果
7. **经验法则**：他明确说出的教训或建议（直接引用原话）

**信息不完整时的处理：**
- 有问题 + 有方案 + 有教训 → 完整记录
- 有问题 + 有方案，没有教训 → 记录，教训标注（可从方案推断）
- 只有教训，没有具体事件 → 归入 work_analyzer 的经验知识库，不作为独立经验
- 碎片太少，无法还原过程 → 标注（原材料不足，建议向本人确认）
